WP-01 · Open Standards Gap · Intent Management / API Design

PRAGMA

From Intent Expression to System Action: Designing a REST API Contract for Declarative Intent Management
Status Draft V1
Date 2026-05-11
Author Sumit Kumar Mitra

Abstract

A growing class of open standards for distributed system management defines rich information models for intent — structured declarations of desired system state. These models specify what operators want: objectives, constraints, targets, and fulfilment conditions. Yet the specifications consistently stop at the boundary where they become useful: no REST API binding exists that makes intent information models machine-callable across vendor and platform boundaries.

This paper proposes PRAGMA — a normative REST API design that bridges declarative intent information models to an OpenAPI 3.1 resource contract. PRAGMA introduces a four-resource model, a negotiation workflow as a REST state machine, and a cross-standard comparison that surfaces both alignment opportunities and irreconcilable gaps requiring new standardisation work.

1. The Problem: Information Models Without Protocols

1.1 What Declarative Intent Models Define

Modern distributed infrastructure management is converging on intent-based paradigms: instead of specifying how to configure a system, operators specify what outcome they require and allow the management plane to determine the implementation. Multiple open standards bodies have independently developed formal vocabularies for this pattern.

The core information elements are structurally consistent across these models:

ElementRole
IntentExpectationFormal objective — goals, requirements, constraints
IntentObjectContainer holding the complete intent definition
ConditionConstraints on fulfilment (temporal, resource, policy)
ContextOperational scope — time, location, policy domain
ExpectationTargetTarget resources or services the intent applies to
FulfilmentInfoActual fulfilment status and deviation detail
IntentObjectInstanceRuntime instantiation — current system state against the intent

1.2 What These Models Do Not Define

Existing intent specifications explicitly scope themselves to requirements and information model only. They state no REST endpoints, no HTTP verbs, no request/response schemas, no error codes, no pagination conventions, no authentication binding.

The consequence is concrete: a system implementing Vendor A's management plane cannot programmatically exchange intent objects with a system implementing Vendor B's, without a proprietary adapter. No conformance test suite can exist. No open-source reference implementation is possible. The ecosystem is blocked at the last metre — the wire format.

1.3 The Missing Protocol Binding Layer

Every major intent standard has a corresponding API specification gap. The pattern is consistent: a rich information model with no REST binding. PRAGMA fills this gap with a design that is normatively grounded in existing information models while being independently implementable by any platform.

2.1 Existing Intent APIs

Service management bodies have published intent APIs with REST CRUD endpoints and lifecycle states. These are loosely typed — expressions support natural language — and critically lack intent negotiation: the bilateral feasibility exchange that must precede commitment at the infrastructure layer.

2.2 Infrastructure-Layer Intent Models

Infrastructure management standards define rigorous intent models with formal IntentExpectation structures, FulfilmentInfo monitoring, and negotiation semantics. But they stop at the information model — no REST binding, no wire format, no conformance surface. A PRAGMA API designed against these models could also serve as a cross-domain translation layer — the foundation for companion paper ROSETTA (WP-07).

2.3 The Convergence Opportunity

The structural similarity across independently developed models suggests a universal binding is achievable. PRAGMA is designed to be that binding: drawing REST conventions from Layer 1 and schema rigour from Layer 2 to produce a single specification that any compliant implementation can conform to.

3. PRAGMA — Proposed REST API Architecture

3.1 Design Principles

  • Information-model normative: Every resource attribute maps 1:1 to a defined information element — no semantic gaps, no extensions that break conformance
  • REST convention compliant: Follows established REST API patterns — pagination, filtering, RFC 7807 problem detail error model, subscription/notification patterns
  • Cross-standard aligned where possible: Resources named to facilitate translation across competing standards
  • OpenAPI 3.1: Full schema definition enabling auto-generation of client/server stubs and conformance test tooling

3.3 Intent Lifecycle as a REST State Machine

POST /intents
     │
     ▼
  [Created]
     │  POST /intents/{id}/negotiate
     ▼
  [Negotiating]
     │  negotiation result: feasible
     ▼
  [Fulfilling]  ◄──── PATCH /intents/{id} (update)
     │
     ├──► [Fulfilled]   — GET /intents/{id}/fulfilment: full compliance
     ├──► [Degraded]    — FulfilmentInfo.deviations populated; notification sent
     └──► [Failed]      — DELETE or PATCH to re-attempt

3.5 Negotiation Workflow

The negotiation sub-resource models the bilateral feasibility exchange as an asynchronous REST interaction. The consumer submits an intent and constraints; the management plane evaluates feasibility and returns either a confirmation or a set of proposed modifications the consumer can accept or reject.

// Request
POST /intents/{intentId}/negotiate
{
  "negotiationMode": "async",
  "constraints": {
    "deadline_utc": "2026-05-15T00:00:00Z",
    "acceptable_degradation_pct": 5
  }
}

// Response (202 Accepted)
Location: /intents/{intentId}/negotiate/{negotiationId}

// Callback result
{
  "negotiationId": "uuid",
  "feasibility": "PARTIAL",
  "proposedModifications": [{
    "expectationId": "exp-01",
    "metric": "latency_ms",
    "originalValue": "<5",
    "proposedValue": "<8",
    "reason": "Insufficient compute headroom on target cluster-03"
  }],
  "expiresAt": "2026-05-11T18:00:00Z"
}

5. Cross-Domain Extension — ROSETTA Preview

A single management domain's intent scope terminates at its boundary. When a system spans multiple administrative domains, intents must be propagated and translated across those boundaries. PRAGMA anticipates this by including domainId in ExpectationTarget, defining intentOriginRef for forwarded intents, and making negotiation results portable. Full treatment in companion paper ROSETTA (WP-07).

6. Implementation Path

6.1 Reference Implementation

PRAGMA can be prototyped using FastAPI (Python) for OpenAPI 3.1 auto-generation, Kubernetes Custom Resource Definitions for IntentObject with a controller implementing the state machine, and CNCF CloudEvents for portable notification payloads with zero platform dependency.

8. Conclusion

Declarative intent management is a well-designed paradigm that has been waiting for its protocol binding. The absence of a REST API layer is not theoretical — it is the concrete reason that intent-based system automation remains vendor-specific in 2026 rather than an interoperable industry capability.

The automation industry does not need another intent framework whiteboard. It needs the API that makes intent executable. That is PRAGMA.
